# Module: Generator

**Invoked by**: `skill.md` when mode is `generate`

**Purpose**: Accept a UIR document and produce ecosystem-specific configuration
files for one or more target ecosystems. Uses templates from `templates/`.

---

## Inputs

- **UIR source**: Either a file path to `uir.json` or inline JSON provided by the user.
- **Target ecosystems**: One or more of `claude-code`, `cursor`, `codex`, `aider`.
  Default: all four.
- **Output directory**: Where to write files. Default: same directory as UIR file,
  or CWD if UIR was inline.
- **Overwrite mode**: `ask` (default), `force`, `skip-existing`.

---

## Pre-Generation Validation

Before generating any files:
1. Confirm the UIR conforms to `schema/uir.schema.json` (check required fields: meta, context, rules).
2. If `rules` is empty, warn: "No rules found in UIR. Generated configs will have minimal behavioral guidance."
3. If `context.project_summary` is missing or under 20 characters, warn.
4. Continue generation regardless of warnings (don't block on WARNING-level issues).

---

## Template Variable Substitution

Templates use `{{VARIABLE}}` placeholders. Replace them as follows:

| Placeholder | UIR source |
|---|---|
| `{{PROJECT_NAME}}` | `meta.name` |
| `{{PROJECT_SUMMARY}}` | `context.project_summary` |
| `{{TECH_STACK_LIST}}` | `context.tech_stack[]` formatted as a markdown list |
| `{{DIRECTORY_LAYOUT}}` | `context.directory_layout` formatted as a markdown table |
| `{{BUILD_COMMANDS}}` | `context.build_commands` formatted as a code block |
| `{{ENV_VARS_TABLE}}` | `context.environment_variables[]` as a markdown table |
| `{{ENTRY_POINTS}}` | `context.entry_points[]` as a bullet list |
| `{{GENERATED_DATE}}` | `meta.generated_at` formatted as YYYY-MM-DD |
| `{{UIR_VERSION}}` | `meta.version` |

For any placeholder with no UIR data: replace with the placeholder wrapped in a
comment: `<!-- TODO: fill in {{PLACEHOLDER}} -->` for markdown files, or
`# TODO: fill in PLACEHOLDER` for YAML/JSON files.

## Generation: `claude-code`

Output directory: `<output>/.claude-code/` (or the repo root if generating in-place).

### File: `CLAUDE.md`

Use `templates/claude-code/CLAUDE.md.tmpl`.

Structure:
1. Header comment: `<!-- Generated by cross-skills {{GENERATED_DATE}} — edit uir.json to regenerate -->`
2. `# {{PROJECT_NAME}}` — project name as H1
3. `## Project Overview` — `context.project_summary`
4. `## Tech Stack` — table from `context.tech_stack[]`
5. `## Directory Layout` — table from `context.directory_layout`
6. `## Build & Dev Commands` — code block from `context.build_commands`
7. `## Environment Variables` — table from `context.environment_variables[]`
8. `## AI Behavior Rules` — one `###` section per `rules[].category`, rules listed as bullets
   - For rules with `rationale`: include as italic sub-bullet
   - For rules with `examples.good`/`examples.bad`: include as fenced code blocks
   - Sort within each category by `priority` ascending (lowest = most important first)
   - Include only `scope: "global"` and `scope: "language"` rules here; file-glob rules go to sub-files
9. `## File-Specific Rules` — for each unique `scope_value` in file-glob rules:
   - `### <glob pattern>` heading
   - List all rules for that glob
10. `## Agent Definitions` — render `agents[]` if present; link to AGENTS.md
11. `## Memory & Context` — list `memory.always_load[]` files and their purpose
12. `## Hooks` — describe each hook in `hooks[]` as a bullet
    - Note: hooks are implemented in `settings.json`, this section is documentation only

### File: `AGENTS.md`

Use `templates/claude-code/AGENTS.md.tmpl`.

Only generate if `agents[]` is non-empty. Structure:
1. Header comment
2. `# Agents` — H1
3. For each agent in `agents[]`:
   - `## <agent.name>` — H2
   - `**Role**: <agent.role>`
   - `**Scope**: <agent.scope>` (if set)
   - Instructions body
   - `**Allowed tools**: <agent.tools_allowed[]>` (if set)

### File: `settings.json`

Use `templates/claude-code/settings.json.tmpl`.

Structure (JSON):
```json
{
  "permissions": {
    "allow": [ ...permissions.allow ],
    "deny":  [ ...permissions.deny ]
  },
  "hooks": [
    { "event": "<event>", "matcher": "<matcher>", "hooks": [{ "type": "command", "command": "<command>" }] }
  ],
  "env": {
    "<key>": "<value>"
  }
}
```
- `permissions.ask_on[]` → Claude Code has no native ask_on in settings.json; convert each entry to a `PreToolUse` hook that echoes a confirmation prompt. Add `# CROSS-SKILLS: ask_on converted to hook` comment.
- If `permissions.allow` and `permissions.deny` are both empty: omit the `permissions` key.
- If `hooks[]` is empty: omit the `hooks` key.
- Only include `env` if `context.environment_variables[]` has entries with `example` values.

### Skill files (one per skill in `skills[]`)

Use `templates/claude-code/skill.md.tmpl` for each entry in `skills[]`.

Preferred layout (cross-tool content.md approach):
1. Write shared instructions to `.ai/skills/<skill-id>/content.md` (no frontmatter).
2. Write `.claude/skills/<skill-id>/SKILL.md` with Claude Code frontmatter only.
3. Create symlink: `.claude/skills/<skill-id>/content.md` → `../../../.ai/skills/<skill-id>/content.md`
   - Linux/Mac: `ln -s ../../../.ai/skills/<skill-id>/content.md .claude/skills/<skill-id>/content.md`
   - Windows: `mklink .claude\skills\<skill-id>\content.md ..\..\..\ai\skills\<skill-id>\content.md`

Flat layout (legacy fallback, single harness only): write `.claude/skills/<skill-id>.md`
combining frontmatter + body. This is still valid but does not support cross-tool sharing.

## Generation: `cursor`

Output directory: `<output>/.cursor/rules/`

### Cursor Rule File Strategy

Split UIR rules into Cursor `.mdc` files by scope and priority:

| UIR rules | Cursor file | alwaysApply | globs |
|---|---|---|---|
| scope=global, priority ≤ 3 | `00-critical.mdc` | true | none |
| scope=global, priority 4–7 | `01-conventions.mdc` | true | none |
| scope=global, priority 8–10 | `02-suggestions.mdc` | false | none |
| scope=file-glob | `10-<sanitized-glob>.mdc` | false | the glob value |
| scope=language | `20-<language>.mdc` | false | language-specific glob |
| scope=task-type | `30-<task-type>.mdc` | false | none |
| skills[] (translated) | `50-skill-<id>.mdc` | false | skill.scope or none |

### Root index file: `.cursor/rules/index.mdc`

Use `templates/cursor/index.mdc.tmpl`. Always `alwaysApply: true`. Contains:
- Project summary
- Tech stack list
- Entry points
- Links to other rule files (as comments)

### Per-file generation

For each `.mdc` file:
1. Use the appropriate template (`rule-always.mdc.tmpl`, `rule-glob.mdc.tmpl`, or `rule-manual.mdc.tmpl`)
2. Frontmatter:
   ```yaml
   ---
   description: <category or purpose of rules in this file>
   alwaysApply: <true|false>
   globs: [<pattern>]  # omit if no globs
   ---
   ```
3. Body: rules as a numbered or bulleted list in plain English
4. Include rationale as `> **Why**: <rationale>` blockquotes when present

### Degradation annotations for Cursor

- `permissions.*` → append to `00-critical.mdc` as: `# CROSS-SKILLS: DEGRADED — permissions converted to rule text (no enforcement)\n- Do not run: <deny pattern>`
- `hooks[]` → append to `01-conventions.mdc` as: `# CROSS-SKILLS: DEGRADED — hook converted to rule text (not executed)\n- Before <matcher>: always <description>`
- `agents[]` → prepend to `index.mdc` as a "Agent Context" section: `# CROSS-SKILLS: DEGRADED — sub-agents flattened to context (no orchestration)`
- `extensions.cursor` overrides → merge last, overwriting any conflicting values

## Generation: `codex`

Output directory: `<output>/.codex/`

### File: `AGENTS.md`

Use `templates/codex/AGENTS.md.tmpl`. OpenAI AGENTS.md spec format.

Structure:
1. `# Project: {{PROJECT_NAME}}` — H1
2. `## Context` — project summary, tech stack, directory layout, build commands
3. `## Instructions` — all global rules as a flat numbered list (no scope grouping)
   - Note: file-glob scoping is lost → add `# CROSS-SKILLS: DEGRADED — file-glob rules flattened; apply these when relevant`
4. For each agent in `agents[]`:
   - `## <agent.name>` — H2
   - `**Role**: <role>`
   - `**Instructions**: <instructions>`
   - `**Tools**: <tools_allowed>` (if set)
5. If no agents: include a default `## Assistant` section with the global instructions

### File: `system-prompt.txt`

Use `templates/codex/system-prompt.txt.tmpl`. Flat prose (no markdown headings).

Content:
- Paragraph 1: project summary
- Paragraph 2: tech stack as prose
- Paragraph 3+: rules as prose paragraphs grouped by category

### Skill files for Codex (`.agents/skills/`)

For each skill in `skills[]`, also generate the cross-tool standard layout:
1. Write `.agents/skills/<skill-id>/SKILL.md` with **only** `name` + `description`.
   - Codex rejects unknown frontmatter fields with an error — never add other fields here.
   - The `description` must be self-contained (covers what + when); Codex does not read `when_to_use`.
2. Create symlink if shared source exists: `.agents/skills/<skill-id>/content.md` → `../../../.ai/skills/<skill-id>/content.md`
   - If no `.ai/skills/` source exists yet, write instructions inline in `content.md` (no symlink).

### Degradation annotations for Codex

- `hooks[]` → `# CROSS-SKILLS: NOT TRANSLATABLE — lifecycle hooks have no Codex equivalent; dropped`
- `permissions.*` → `# CROSS-SKILLS: DEGRADED — permissions converted to rule text`
- `memory.always_load` → mention files as "always consider these files for context" in Instructions
- `skills[]` → append as `## Available Tasks` section in AGENTS.md AND generate `.agents/skills/` directory files (above)

## Generation: `aider`

Output directory: `<output>/.aider/`

### File: `.aider.conf.yml`

Use `templates/aider/aider.conf.yml.tmpl`.

```yaml
# Generated by cross-skills {{GENERATED_DATE}}
model: <extensions.aider.model or "claude-sonnet-4-5">
read:
  - CONVENTIONS.md
  <...memory.always_load entries...>
auto-commits: <extensions.aider.auto_commits or true>
dirty-commits: <extensions.aider.dirty_commits or true>
```

### File: `CONVENTIONS.md`

Use `templates/aider/CONVENTIONS.md.tmpl`.

Structure: one `##` section per rule category. Within each section, rules as a
numbered list. Rationale included as italic sub-items.

Note at top: `<!-- This file is read by Aider via .aider.conf.yml. Edit uir.json and regenerate to update. -->`

### File: `.aiderignore`

Use `templates/aider/aiderignore.tmpl`.

One pattern per line from `ignore_patterns[]` + `memory.ignore[]` (deduplicated).
Preceded by a comment header.

### Degradation annotations for Aider

- `hooks[]` → `# CROSS-SKILLS: NOT TRANSLATABLE — hooks have no Aider equivalent` in CONVENTIONS.md header
- `permissions.*` → convert deny list to CONVENTIONS.md rule: `- Never run: <pattern>`
- `agents[]` → flatten each agent to a `## <agent.name> Role` section in CONVENTIONS.md
  with `# CROSS-SKILLS: DEGRADED — sub-agents flattened to conventions (no orchestration)`
- `scope: "file-glob"` rules → flatten, add `(applies to: <glob>)` note to rule text
- `skills[]` → render each as a named section `## Skill: <skill.name>` in CONVENTIONS.md

## Generation: `copilot`

Output directory: `.github/skills/`

GitHub Copilot reads skill files from `.github/skills/<skill-id>/SKILL.md`. It uses
only `name` and `description` frontmatter; all other fields are silently ignored.

### Skill files for Copilot

For each skill in `skills[]`:

1. Write `.github/skills/<skill-id>/SKILL.md`:
   ```yaml
   ---
   name: <skill-id>
   description: >
     <what it does and when to use it — self-contained; Copilot reads only name+description>
   ---
   ```
   Do not add `when_to_use`, `allowed-tools`, or any other field — they are ignored and
   create noise.

2. Create symlink if shared source exists: `.github/skills/<skill-id>/content.md` → `../../../.ai/skills/<skill-id>/content.md`
   - If no `.ai/skills/` source exists, write instructions directly in `content.md` (no symlink).

### Degradation annotations for Copilot

- `permissions.*` → `# CROSS-SKILLS: NOT TRANSLATABLE — Copilot has no permissions concept; dropped`
- `hooks[]` → `# CROSS-SKILLS: NOT TRANSLATABLE — Copilot has no lifecycle hooks; dropped`
- `agents[]` → `# CROSS-SKILLS: NOT TRANSLATABLE — Copilot has no sub-agent definitions; dropped`
- `rules[]` → Copilot does not have a global rules/conventions mechanism equivalent to CLAUDE.md;
  consider adding project-level context via `.github/copilot-instructions.md` (out of scope for
  skill generation — note in output).

## Output Manifest

After all files are written, print:

```
## Generated Files

### Shared skill sources
- .ai/skills/<id>/content.md            (one per skill — shared instructions)

### claude-code

- AGENTS.md                          (if agents present)
- settings.json                      (if permissions or hooks present)
- .claude/skills/<id>/SKILL.md       (one per skill — Claude Code frontmatter)
- .claude/skills/<id>/content.md     (symlink → shared source)

### cursor
- .cursor/rules/index.mdc
- .cursor/rules/00-critical.mdc
- .cursor/rules/01-conventions.mdc
- ...
- .cursor/skills/<id>/SKILL.md       (one per skill — Cursor frontmatter)
- .cursor/skills/<id>/content.md     (symlink → shared source)

### codex
- AGENTS.md
- system-prompt.txt
- .agents/skills/<id>/SKILL.md       (one per skill — name+description only)
- .agents/skills/<id>/content.md     (symlink → shared source)

### copilot
- .github/skills/<id>/SKILL.md      (one per skill — name+description only)
- .github/skills/<id>/content.md    (symlink → shared source)

### aider
- .aider.conf.yml
- CONVENTIONS.md
- .aiderignore

## Degradation Warnings
- <N> fields degraded (see CROSS-SKILLS: DEGRADED comments in output files)
- <N> fields dropped (see CROSS-SKILLS: NOT TRANSLATABLE comments)

Next: run `cross-skills validate` on the output files to catch any issues.
```

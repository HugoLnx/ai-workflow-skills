# Rules Builder

How to build rules from `.ai/rules/` to harness output files, generate build scripts, and validate/list rules.

**Also read**: `references/rules-frontmatter-expert.md` when writing or validating frontmatter fields.

---

## Running the Existing Build Script

```bash
python .ai/skills/multi-ai/scripts/build-rules.py
```

The script is idempotent — safe to run multiple times. It prints a summary of files created, updated, and unchanged. A non-zero exit means at least one error.

---

## Build Algorithm

```
For each .ai/rules/*.md file:

  1. Determine slug = filename without .md extension
  2. Check for matching .ai/rules/<slug>.yml sidecar

  3. Write .cursor/rules/<slug>.mdc:
     - If sidecar has cursor section: use cursor.description, cursor.globs, cursor.alwaysApply
     - If no sidecar or cursor section: emit alwaysApply: true, omit globs
     - Append generated-by comment, then verbatim .md content
     - Skip write if content unchanged (compare before writing)

  4. Write .github/instructions/<slug>.md:
     - If sidecar has copilot.applyTo: emit applyTo frontmatter block
     - If no copilot.applyTo: omit frontmatter entirely
     - Append generated-by comment, then verbatim .md content
     - Skip write if content unchanged

  5. Update managed imports block in CLAUDE.md and AGENTS.md:
     - Find <!-- cross-rules:imports:start --> … <!-- cross-rules:imports:end --> delimiters
     - Replace block content with @file .ai/rules/<slug>.md entries for all .md files
     - If delimiters absent: append the block at end of file
     - Never touch content outside the delimiters

  6. Print summary: files created, updated, unchanged
```

## Generating a Build Script

When no build script exists, or the user asks for one:

1. Ask which runtime: **Node.js** (default), Python, Ruby, or shell
2. Ask where to save (default: `scripts/build-rules.<ext>`)
3. Generate a self-contained script implementing the algorithm above
4. Use only the standard library — no third-party packages
5. Make executable (`chmod +x`) on Unix
6. If the project has `package.json` and runtime is Node.js: offer to add `"build:rules"` to `scripts`

## Validate Operation

Consult `references/validator.md` for the full check list and report format.

Rules-specific checks summary:
- Orphaned sidecars (`.yml` with no matching `.md`)
- Invalid YAML (parse errors in `.yml` files)
- Master root line counts (`CLAUDE.md`, `AGENTS.md`)
- Multiple rule files (prefer consolidating)
- Stale generated files (`.cursor/rules/*.mdc` or `.github/instructions/*.md` out of sync)
- Stale import blocks (managed block out of sync with `.ai/rules/`)
- Code style in rules (ESLint, Prettier, Biome, Ruff prose)
- Conditional content in rules (context-specific language)

## List Operation

Print a table of all `.ai/rules/*.md` files:

```
Rule              Always-on   Cursor globs              Copilot applyTo   Built files exist?
────────────────────────────────────────────────────────────────────────────────────────────
root              yes         —                         —                 yes
api-conventions   no          src/api/**/*.ts           src/api/**        yes
testing-policy    yes         —                         —                 no (stale)
```

Derive each column from the `.yml` sidecar (or always-on defaults if no sidecar). Check file existence for the last column.

---

## Generated File Comments

Every generated file includes a comment immediately after the frontmatter block:

**Cursor `.mdc`**:
```
<!-- generated by build-rules script — edit .ai/rules/<name>.md to update -->
```

**Copilot `.md`**:
```
<!-- generated by build-rules script — edit .ai/rules/<name>.md to update -->
```

If the user edits a generated file directly and re-runs the build script, their edits will be overwritten. Always redirect edits to `.ai/rules/<name>.md`.
